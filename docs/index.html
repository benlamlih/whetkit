THE PROBLEM

Too many tools.
Too many that look alike.

MCP servers grow. They expose dozens of overlapping, ambiguously-named tools. The agent has to guess which one the task needs — and it guesses wrong.

One wrong pick cascades: a bad call, a confused retry, a failed task. You feel it as flaky agents. It's actually a tool-selection problem you can measure.

HOW IT WORKS

Point it at a server and a set of tasks. Get a curated tool set back.

01

Point & run

Give it your MCP server and a set of eval tasks. It runs the agent against every task, exactly as production would.

02

Score & trace

It scores tool-selection hit-rate and captures the full reasoning-path trace for every task — so you see why a tool was chosen.

03

Curate & report

It prunes, renames, and rewrites tools, then reports the before/after improvement.

◆ Curation is non-destructive — it's an overlay that sits in front of your server. Your source tools are never touched.
QUICKSTART

A real command, not a dashboard.

Install once, then inspect your server, run the eval, and curate. Everything runs locally against your MCP server.

Read the docs ↗
whetkit — zsh
# install
$ uv tool install whetkit
# inspect the server's tools
$ whetkit inspect --server ./server.json
↳ 48 tools · 9 flagged ambiguous
# run the eval tasks
$ whetkit run --server ./server.json --tasks ./tasks
↳ hit-rate 61% over 100 tasks
# curate + report
$ whetkit curate --server ./server.json --tasks ./tasks
↳ pruned 31 · renamed 5 · rewrote 8
↳ hit-rate 61% → 89% · report.html
